# Atlas AI OS — правила для агентов

Этот файл действует для всего репозитория. Более близкий `AGENTS.md` может уточнять правила только для своего поддерева, но не отменяет ограничения безопасности и ручные gates ниже.

## Назначение продукта

- Atlas — внутренняя многопользовательская контент-фабрика владельца для роста и монетизации аудитории в социальных сетях.
- Это не SaaS: самостоятельная регистрация внешних клиентов, тарифы, подписки и SaaS-биллинг не входят в продукт.
- Помощники работают по ограниченным ролям. `created_by` нельзя считать надёжной границей владельца или workspace.
- Приоритеты: рост аудитории, качество и регулярность контента, конверсия в монетизацию и стоимость производства.

## Обязательный контекст перед работой

1. Получить актуальный `main` и записать его SHA.
2. Проверить все открытые PR и возможные пересечения файлов.
3. Прочитать `docs/PROJECT_STATE.md`.
4. Прочитать относящиеся к задаче контракты в `docs/architecture/`.
5. Подтвердить назначенную область и узкий scope задачи.

Не полагаться на память старого чата. GitHub, `docs/PROJECT_STATE.md`, архитектурные контракты и фактический код являются источниками истины.

## Области и координация

- `00` — координатор: архитектура, приоритеты, roadmap, распределение задач и контроль PR.
- `01` — AI-модели и Character Brain.
- `02` — сцены, референсы, Modal и кэш.
- `03` — контент-фабрика, lifecycle, публикации и аналитика.
- `04` — интерфейс, адаптивность и UI-модули.
- `05` — Supabase, Storage, RLS, Vercel, auth и инфраструктура.

Работать только в назначенной области. Межобластное изменение сначала согласовать с областью `00`. Не менять чужие статусы в `PROJECT_STATE.md` и не расширять задачу ради попутного рефакторинга.

Не запускать параллельные изменения в одних и тех же файлах. Особенно осторожно работать с `src/app/dashboard.tsx`, общими типами, migration-файлами и `docs/PROJECT_STATE.md`.

## Рабочий процесс

1. Начать от актуального `main` с чистым working tree.
2. Создать отдельную ветку `agent/<краткая-задача>`. Управляемая Codex Cloud ветка `codex/<...>` допустима, если имя задаёт платформа.
3. Вносить минимальные изменения только в согласованный scope. Сохранять чужие и несвязанные изменения.
4. Не скрывать ошибки, не отключать ESLint/TypeScript и не ослаблять проверки ради зелёного CI.
5. Выполнить проверки, соответствующие риску задачи.
6. После существенного изменения обновить только свою строку и одну запись журнала в `docs/PROJECT_STATE.md`; журнал содержит не более 15 записей.
7. Создать draft PR с целью, файлами, проверками, рисками, расходами API/GPU и следующим шагом.
8. Дождаться GitHub Actions и Vercel Preview. Агент не выполняет merge: финальное слияние подтверждает владелец или координатор вручную.

Codex Cloud публикует изменения через кнопку **Create draft PR** в интерфейсе. Не добавлять GitHub token в repository secrets и не обходить отсутствие shell `origin` или `gh` самодельной авторизацией.

## Проверки

Минимум перед draft PR:

```bash
npm ci
npm run build
git diff --check
git status --short
```

- При изменении migrations также должен пройти существующий Supabase migration CI и pgTAP.
- Docker и Supabase CLI могут отсутствовать локально; не имитировать успешный прогон. Использовать существующий изолированный GitHub Actions workflow.
- Существующий lint baseline не скрывать. Если `npm run lint` падает из-за известных ошибок, отделить новые ошибки от baseline и честно сообщить результат.
- Предупреждения Node/Supabase `EBADENGINE` не игнорировать и не исправлять попутно вне отдельной задачи.
- Не запускать Modal GPU или OpenAI API для проверки сборки, документации, UI или схемы.

## Жёсткие gates безопасности

### Production и Supabase

- По умолчанию production-доступ запрещён.
- Read-only production-проверка требует явного разрешения владельца и обезличенного результата.
- Запрещены без отдельного ручного gate: `supabase link`, `db push`, `migration repair`, production DDL/DML, изменение RLS/Storage, service-role операции и запись credentials.
- `202607120600_core_schema_baseline.sql` предназначен только для чистой среды и никогда не запускается на существующем production.
- Production migration-history reconciliation остаётся запрещённым до выполнения gates из `docs/architecture/PRODUCTION_MIGRATION_RECONCILIATION.md`.
- Никогда не публиковать secrets, connection strings, email, пользовательские UUID, prompts, содержимое записей или Storage paths в логах и PR.

### OpenAI, Modal и публикация

- Не запускать платные OpenAI/Modal операции без необходимости задачи и явного согласования бюджета.
- По умолчанию генерировать один результат. Повторы должны иметь idempotency, cache preflight, лимит и понятную стоимость.
- Не выполнять реальную публикацию в социальные сети без ручного approval владельца.
- Approval контента, render status и delivery status — разные состояния; одно не подменяет другое.

### Данные и роли

- Любой новый cache, idempotency key и lookup должен быть tenant/workspace scoped.
- Не использовать legacy `created_by` как автоматический `owner_id`.
- Изменения ownership/RLS выполняются additive и по отдельному согласованному cutover-плану без потери данных.

## Правила review

При проверке PR в первую очередь искать:

- production writes или обход ручных gates;
- cross-tenant доступ, cache leakage и смешение `created_by`/`owner_id`;
- повторные OpenAI, GPU или publishing вызовы без idempotency;
- проглоченные Supabase/API ошибки и ложный успешный UI;
- расхождение runtime, migrations и подтверждённой production schema;
- изменение DOM/CSS или поведения вне заявленного scope;
- отсутствие build/CI evidence и необновлённое состояние проекта.

## Definition of done

Задача завершена, когда:

- scope соблюдён и чужие функции не сломаны;
- релевантные проверки выполнены и результаты указаны без сокрытия предупреждений;
- remote diff содержит только ожидаемые файлы;
- draft PR создан, CI/Vercel проверены, production и расходы описаны;
- `docs/PROJECT_STATE.md` обновлён после существенного изменения;
- указан один следующий безопасный шаг.
